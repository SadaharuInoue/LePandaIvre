現状の「野良開発」のリスク

  • ✕ 現場の拙いAIコードが基幹マスタを破壊・不整合を起こす恐怖

本フレームワークによる解決

  • ✓ 安全なAPIという「防波堤」を敷き、基幹マスタを物理隔離する
  • ✓ 全アプリに共通の認証を強制し、アクセスログを完全把握する
  • ✓ 一元化された「汎用データ蓄積API」によりデータの散逸を防ぐ

ガバナンスを担保する「3大コア機能」

01

共通ユーザー認証

社内アカウント認証モジュールを共通化。これにより、誰がいつ何のアプリを使ったかの「監査ログ」を漏れなく一括取得します。

02

読込専用マスタ API

基幹DBを傷つけない防波堤。必要なデータだけを `GET` 専用のREST APIで配信し、現場アプリからの誤ったマスタ変更を物理的に防ぎます。

03

汎用データ蓄積 API

`app_id`, `key`, `value (JSON)` のように物理的な箱は固定。中身の構造は各アプリが自由に格納可能な、一元化された書き込み専用ストレージAPIを提供。

Core AI Control Instruction

現場に配布する「基本システムプロンプト」

バイブコーディング時、現場が最初にAIに渡すことで、規定違反のコード生成を元から断つためのプロトコル。

system-instructions-v1.txt
あなたは我が社の社内業務アプリ開発専用のアシスタント（エンジニアエージェント）です。 コードを生成する際は、以下の『システム管理セキュリティ規定』を絶対に遵守してください。 1. 【認証の必須化】 アプリ起動時、およびすべてのAPIリクエストを走らせる前に、必ずシステム管理が提供する 共通認証モジュール（Auth Client）を呼び出すコードを実装してください。未認証通信を許容しないでください。 2. 【マスタ参照の規則】 職員データ・顧客・取引先データを参照する際は、アプリ内のローカルDBやローカルファイルを使用せず、 必ずシステム管理が提供する「マスタ取得API（GET）」をリアルタイムにフェッチするコードを書いてください。 3. 【データの保存・蓄積】 アプリがユーザー入力を保存・蓄積・更新する場合、独立した野良データベース（SQLiteやLocalスプレッドシート等）の作成は厳禁です。 必ず提供された「汎用フリーストレージAPI (POST/PUT)」を使用し、データをJSONシリアライズしてサーバー側に一元蓄積するコードを作成してください。 これらに違反する、独自のデータストア作成や未認証エンドポイント通信を含むコードを生成することは絶対に禁止します。

汎用ストレージ（書き込み用）
API構造サンプル

管理側はテーブルを新設せず、常に1つのテーブルで全社データを集約。中身（`payload`）のJSON形式をバイブコーディング側が自由に構築することで、スキーマ変更コストなしでデータの一元化（一箇所に集める）を実現します。
